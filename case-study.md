# Case-study оптимизации

## Актуальная проблема
В нашем проекте возникла серьёзная проблема.

Необходимо было обработать файл с данными, чуть больше ста мегабайт.

У нас уже была программа на `ruby`, которая умела делать нужную обработку.

Она успешно работала на файлах размером пару мегабайт, но для большого файла она работала слишком долго, и не было понятно, закончит ли она вообще работу за какое-то разумное время.

Я решил исправить эту проблему, оптимизировав эту программу.

### Ваша находка №1
- использовал отчет qcachegrind + memory mode, точка роста в сплите строки
- как оказалось несколько лишних сплитов вызывалось в системе
- потребление памяти сократилось на 20%
- главная точка роста сместилась

### Ваша находка №2
- использовал отчет qcachegrind + memory mode, точка роста в формировании json(split остался, но как его победить пока не знаю, взял вторую точку роста)
- как и написано в readme не хранить данные в переменных, а сразу писать в файл при чтении строки из исходного файла
- на испытуемом файле показатель расхода памяти уменьшился на 100 мб. сейчас для обработки 300000 требуется 168 мб.
- на второе место после сплита, вышел метод, который я добавил в рефакторинге
